# Foundry 项目功能与能力说明

本文档基于当前代码库结构（`apps/*`、`infrastructure/*`、`contracts/*`）做**已实现能力**的归纳，便于快速理解系统能做什么、适合解决哪些问题。

---

## 1. 项目定位

**Foundry** 是一套基于 **NestJS + pnpm monorepo（Turbo）** 的微服务风格后端：对外由 **网关** 统一入口，**应用 API** 承载业务与数据访问，**Worker** 异步消费消息并编排长任务与 AI 流程，**Webhooks** 服务接收与配置出站回调。底层使用 **PostgreSQL**（含迁移与租户 RLS）、消息队列抽象（`@service/messaging`，以 RabbitMQ 等为后端），并提供共享的 **租户、监控、安全、缓存** 等基础设施包。

---

## 2. 服务与职责一览

| 组件 | 职责 |
|------|------|
| **Gateway（`apps/gateway`）** | HTTP 入口、动态路由与 **RPC 转发**、JWT 与多层安全中间件（签名/防重放/CSRF/IP）、限流与熔断、审计、健康与指标、**协作 WebSocket（`/collaboration`）** |
| **API（`apps/api`）** | 领域 REST/RPC 控制面：用户、认证、OAuth、文件、公司、组织、Agent、技能、协作、记忆、任务、计费、模板等；**多租户解析 + Postgres RLS 上下文** |
| **Worker（`apps/worker`）** | 订阅领域事件：登录/用户/公司/组织/Agent/协作/记忆摄入/任务拆解与心跳/计费入账/模板导入等；含 **LangGraph 自治编排（CEO 心跳图等）** 与 Agent 执行适配 |
| **Webhooks（`apps/webhooks`）** | Webhook 接收、配置 CRUD、历史；经网关 RPC 与下游对齐 |
| **contracts** | 跨服务事件契约（`auth` / `user` / `company` / `organization` / `agent` / `collaboration` / `memory` / `task` / `billing` / `template` / `autonomous` 等） |
| **infrastructure** | 消息、租户、迁移、缓存、监控、日志、安全、可选 AI 包等可复用实现 |

## 3. 已实现功能（按业务域）

### 3.1 账户与访问

- **用户**：CRUD、分页查询（经网关 RPC 暴露）。
- **认证**：校验与登录相关能力；**OAuth** 绑定与账号管理。
- **授权模型**：JWT、角色、权限等守卫（API 侧）；网关侧 API Key、IP 策略、限流、熔断。

### 3.2 租户与公司

- **多租户**：请求上下文中的公司与租户信息、`TenantGuard`、**TypeORM + Postgres RLS**（会话级租户隔离）。
- **公司（Companies）**：创建、查询、更新、状态变更等企业生命周期管理。

### 3.3 组织与 Agent

- **组织（Organization）**：组织树、节点、结构变更与审计相关能力（网关已配置对应 RPC 路由）。
- **Agent**：Agent 生命周期、与组织节点联动（如节点迁移同步）、技能绑定与审批等控制面。
- **技能（Skills）**：平台/组织技能管理、执行与日志相关表结构（迁移与模块已存在）。

### 3.4 协作

- **房间 / 消息 / 成员**：协作控制面 API；网关 **Socket.IO** 实时通道，JWT 鉴权后与 RPC 协同。
- **异步**：消息到达、成员进出、部门加入等事件由 Worker 监听处理。

### 3.5 记忆（Memory）

- **检索、文档摄入、摘要** 等 RPC 能力；异步 **ingest** 请求由 Worker 消费。
- 与 **任务完成** 联动：可将任务结果写入公司级记忆（架构与监听器设计已体现）。

### 3.6 任务（Tasks）

- **任务 OS**：仪表盘、拆解、日志等；Worker 侧 **任务拆解**、**心跳 tick**、与 **Autonomous（LangGraph）** 编排联动（预算告警、任务完成自治等监听器）。

### 3.7 计费（Billing）

- **消费记录、预算、模型路由** 等 RPC；事件驱动：**消耗入账**、**任务完成计费**、**心跳刷新预算信号** 等。

### 3.8 模板与商城（Templates / Marketplace）

- **模板预览、导入**；与 **Agent 商城**（列表/购买等）相关的路由与模块；模板导入后 **物化** 由 Worker 幂等处理。

### 3.9 文件与存储

- **文件** 上传/下载/删除；存储抽象支持 **MinIO / S3 / OSS / 本地** 等适配器（以 API 模块实现为准）。

### 3.10 可观测与治理

- **健康检查**、**指标**、**审计日志**（含公司维度扩展）、**链路追踪**、**日志拦截**。
- 网关 **RPC pattern 白名单**、动态路由与路径重写，便于灰度与治理。

## 4. 能解决的需求与典型场景

| 需求类型 | 说明 |
|----------|------|
| **多租户 SaaS 后端** | 公司维度数据隔离、RLS、请求头透传（`x-user-info` / `x-company-id`）与统一网关策略。 |

| **任务编排与自治运行** | 任务拆解、心跳、与 LangGraph 自治流程结合，适合长时运行、需预算与信号联动的自动化。 |
| **用量与成本控制** | 计费记录、预算、模型路由事件，支撑按量计费与预警。 |
| **内容与商业化** | 模板导入与 Agent 商城路径，支撑模板分发与 Agent 交易类场景。 |
| **集成与自动化** | Webhooks 服务 + 消息总线，对接外部系统与内部异步流程。 |

## 5. 使用上的边界说明（诚实范围）

- 本文档描述的是**代码库中已挂载的模块、路由、迁移与事件契约**所体现的能力；具体环境还需配置 **数据库、消息队列、对象存储、AI 密钥** 等（参见 `.env.shared` 与部署说明）。
- **前端客户端**不在 `apps` 核心三件套（gateway/api/worker）的默认交付范围内；本仓库侧重**后端与异步编排**。
- 部分能力（如 **Skills 执行细节、AI 模型选型**）依赖 `infrastructure/ai` 与环境变量，以运行时配置为准。

## 7. 与产品愿景的差距与风险（按重要性）

本节对照「公司创建 → 组织可视化 → Agent 配置 → 实时协作 → 自治运行 → 记忆 → 成本控制 → 模板市场」等产品诉求，归纳**关键差距**；其中部分结论已结合代码结构核对（便于区分「未做」与「仅有骨架」）。

### 7.1 LangGraph + Heartbeat 自治闭环（当前最大差距）

| 维度 | 现状（代码侧） | 风险 |
|------|----------------|------|
| **编排拓扑** | `infrastructure/ai` 中 `buildCeoHeartbeatGraph` 为 **线性流水线**：ingest → plan → validatePersist → summarize → notify，并非 **Hierarchical Supervisor（CEO → 部门主管 → 执行 Agent）** 的多层图。 | 「公司自己跑起来」的核心叙事若依赖层级委派与并行子图，当前实现**不足以单独支撑**。 |
| **Heartbeat** | 存在 `TaskHeartbeatScheduler`、`task.heartbeat.tick` 消费与 `AutonomousOrchestratorService.runHeartbeat` 等路径；CEO 规划前会 RPC **`billing.checkAllowance`**，并在部分路径写入记忆类集合标签。 | **调度粒度、与任务/组织状态的强一致、失败重试与补偿** 仍需产品级设计与更多测试/监控覆盖。 |
| **状态与恢复** | 支持 **LangGraph Checkpointer**（无 DB URL 时退化为 `MemorySaver`，生产需配置 `WORKER_CHECKPOINT_DATABASE_URL` 等）。 | 进程重启、多副本下的 **thread_id 约定、幂等与重复 tick** 需在运维与集成层显式验证。 |

**结论**：自治链路有**可运行的 CEO 心跳图与计费前置校验**，但 **层级监管、执行 Agent 闭环、完整故障恢复** 仍属显著缺口；若不补齐，「自治运行」难以作为稳定卖点。

### 7.2 跨模块深度集成与一致性

| 主题 | 差距说明 |
|------|----------|
| **端到端故事** | 各域多有独立 API/事件，但 **「群聊 → 自动建任务 → 执行 → 写记忆 → 刷新仪表盘 → 计费」** 是否形成**可靠、可测试的一条龙**，在文档与契约层面仍不够透明，需用集成用例与可观测链路固化。 |
| **协作 ↔ 组织 ↔ Agent** | 组织变更、成员/房间事件有监听器，**「动态拉人是否对所有客户端实时一致」** 取决于网关 WS、RPC 缓存与前端订阅策略，后端需明确**事件契约与推送范围**。 |
| **Skills ↔ LangGraph** | Worker 侧存在 **Builtin 工具注册**（如 echo、file-read/code-run 等**占位实现**），与 **数据库中可配置 Skills 的动态注册、权限与审计** 的打通程度有限，需在运行时统一 **ToolRegistry** 与领域技能模型。 |

### 7.3 实时体验与前端支撑

- WebSocket（协作命名空间）已存在，但 **流式输出、审批推送、任务进度、房间生命周期、断线重连与消息序** 等需 **前后端联合约定**；当前仓库以**后端能力**为主，**推送事件清单与版本策略**未在根文档中单列。

### 7.4 商业化与治理能力

| 主题 | 差距说明 |
|------|----------|
| **Billing** | 有记录、路由、预算相关 RPC 与心跳联动；**硬预算拦截、自动降级策略、账单导出/对账** 等是否满足商业化需单独验收。 |
| **Templates / 商城** | 存在导入与幂等 Worker 路径；**「一键物化」是否自动创建完整组织 + Agent + Skills + Memory 分区** 需对照实现逐项勾选。 |
| **权限** | 多租户 + JWT/角色/权限守卫已有；**Owner / Admin / Member 在多公司场景下的细粒度规则** 建议在权限矩阵中单列并配测试。 |

### 7.5 生产就绪与可观测性

- Tracing、Audit、Metrics 等模块存在，但 **自治流程（LangGraph 节点级 span、tick 关联 traceId、失败分支）** 是否 **全链路可查询** 仍需验证。
- 消息侧 **DLQ、重试、自治场景下的毒消息处理** 需在 `messaging` 与各 Listener 上统一策略并文档化。

### 7.6 其他

- **Onboarding**、面向移动端的 **简化 API**、**超大规模**下的组织/Agent/向量检索性能等，多属增强项，需按路线图排期。
