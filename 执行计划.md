### 2. 系统架构建议（2026版升级路线）
当前架构（monorepo + Gateway + API + Worker + Events）**天生适合**可运行数字公司，只是需要**三处战略性加固**：

| 清单阶段 | 当前覆盖度 | 缺失/需硬化点 | 技术选型（2026最新） | 优先级 |
|----------|------------|---------------|----------------------|--------|
| A0 基础数据模型 | 95%（Tenant RLS、Org/Agents/Tasks模型已有） | 完整RunRecord + Artifact不可篡改 | PostgreSQL + Temporal.io（状态机+持久化）+ Event Sourcing轻量版 | ★★★★★ |
| B1-B3 编排层 | 90%（AutonomousModule + Heartbeat + 组织树） | 跨团队工单 + 委派规则引擎 | LangGraph 2.0 + ReAct + Supervisor层 + 技能标签向量路由 | ★★★★ |
| C1-C3 执行与Trace | 80%（Worker执行适配层） | 单次RunRecord全链路 + 工具白名单 + 不可篡改审计 | OpenTelemetry 2.0 + WASM沙箱 + Append-only Audit Log（ClickHouse） | ★★★★★ |
| D1-D3 运营监控 | 70%（MonitoringModule） | 董事会仪表盘 + SLI/SLO + Runbook | Prometheus + Grafana + AI生成的Runbook（Claude 4级） | ★★★★ |
| E1-E3 预算 | 75%（BillingModule） | 硬停止 + 实时计量 + 成本归因 | Redis + RateLimit + billing.checkAllowance原子检查 | ★★★★★ |
| F1-F3 治理审批 | 60%（已有审批队列雏形） | **原子绑定令牌**（未批准→技术上不可执行） | Temporal Workflow + 一次性执行Token（JWT+Redis短命） | ★★★★★ |
| G-H-I | 50% | Supervisor复盘、记忆回灌、多公司模板 | o1-style reasoning + Synthetic Data自进化 + 公司模板导出（JSON Schema + 密钥脱敏） | ★★★ |

**关键trade-off**（我毒舌一下）：
- 别再用“内存Checkpointer”了，生产级长时任务必须上**Temporal.io**（2026年已经彻底取代LangGraph原生checkpointer）。
- 审批不能只是UI队列，必须是**工程闸门**：未批准的执行令牌直接在Worker层被拒绝（零信任）。

### 3. 完整实现方案（按你文档的节奏走）
我们**严格按里程碑竖切**，先把M1-M4打穿，再G自治。每个里程碑交付**一条完整闭环场景**（例如“每日巡检任务”或“发版审批”），贯穿A-F，避免孤岛。

#### **Phase 0：M1（本周可验收）——任务模型 + 心跳 + 全链路ID + 基础仪表盘**
**竖切场景**：CEO Heartbeat自动巡检待办 → 拆解 → 分配 → 执行 → 汇总汇报（用户只在群聊看结果）。

**目录结构调整**（直接可merge）：
```
packages/core/task/
├── domain/          # Task, WorkItem, RunRecord, Artifact（Event Sourcing）
├── application/     # TaskOrchestratorService, HeartbeatScheduler
├── infrastructure/  # TemporalWorkflowAdapter, RunRecordRepository (immutable)
└── presentation/    # TasksRpcController + 董事会Dashboard DTO
```

**核心交付物**：
- 升级Task实体为完整状态机 + DAG依赖（用Temporal Workflow持久化）。
- 每一次run生成唯一`runId`，全链路透传（Gateway → API → Worker → Logging）。
- 董事会仪表盘初版（React Flow + 实时WebSocket推送）。
- 验收标准：一条“每日巡检”任务从创建→heartbeat触发→完成，可在UI/API完整trace（含耗时、工具调用、成本）。

**测试策略**：E2E Cypress + Contract Test（Pact）覆盖事件契约。

**部署**：直接走现有docker-compose + Temporal Worker sidecar。

### 4. 主动优化点（我不会让你踩坑）
- **性能**：RunRecord用ClickHouse Append-only + Materialized View，查询毫秒级。
- **安全**：所有高风险操作（E/F阶段）必须走`execution-token`校验，Token在Redis 30s过期 + 审计强制写入。
- **可观测性**：统一日志字段（`companyId, taskId, runId, agentId, cost`），直接对接现有Logging服务。
- **成本**：M1就接入`billing.checkAllowance`原子检查，防止“自治变账单自治”。

- **M2（下周）**：Run trace + 告警到IM + 故障10分钟定位。
