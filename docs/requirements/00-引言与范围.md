# 0. 引言与范围

## 0.1 文档目的

本文档系列将 **Foundry 后端当前已实现能力** 表述为**可验收的用户功能需求**：谁、在什么场景下、能完成什么、如何验收。它不替代 OpenAPI/Swagger，但为产品、测试与交付提供统一语言。

## 0.2 系统边界（用户可见）

从调用方看，对外主入口为 **API Gateway**（全局 HTTP 前缀一般为 `/api`）。典型路径形态包括：

- **认证**：`/api/auth/*`（注册、登录、刷新、登出、微信 OAuth 等），由网关直接处理。
- **领域 API（v1）**：`/api/v1/*`，由网关按静态/动态路由转发至 **API**、**Webhooks** 或 **Worker** 服务（多数业务为 RPC 至 `apps/api`）。
- **Webhook 接收**：`/api/webhooks/*`（外部系统回调，通常无 JWT）。
- **管理面**：`/api/admin/*`（动态路由、API Key、IP 规则、审计等），需管理员权限。
- **可观测**：如 `/api/metrics`（Prometheus 文本）。
- **实时协作**：WebSocket 命名空间 **`/collaboration`**（详见协作文档与 `collaboration-websocket-contract.md`）。

具体路由以 `apps/gateway/src/modules/routing/config/routes.config.ts` 及动态路由数据为准。

## 0.3 用户角色（简化）

| 角色 | 说明 |
|------|------|
| **访客** | 未登录；仅可访问标注为公开的能力（如注册、登录、部分 OAuth 回调、Webhook 接收）。 |
| **登录用户** | 持有 JWT；可访问租户内业务 API（用户、公司、组织、Agent、协作等），受角色与租户数据隔离约束。 |
| **管理员** | 具备管理角色；可管理用户（如创建用户）、网关管理 API（路由、密钥、IP、审计查询）等，以代码中 `@Roles` 与网关守卫为准。 |
| **集成方 / 运维** | 使用 API Key、签名校验、IP 策略或调用指标端点；配置出站 Webhook 与接收事件。 |

## 0.4 多租户与数据隔离（需求级描述）

- 业务数据按**租户/公司**上下文隔离；API 层通过租户解析与 **PostgreSQL RLS** 会话变量实现隔离（实现细节见 `@service/tenant` 与迁移）。
- **验收关注点**：同一 JWT 下仅能操作当前租户可见资源；跨租户访问应被拒绝（与 Guard/策略一致）。

- 不保证列举每个 DTO 字段；字段级说明以各服务 Swagger（非生产且开启时）为准。
