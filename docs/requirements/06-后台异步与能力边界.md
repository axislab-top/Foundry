# 6. 后台异步、日志与能力边界

## 6.1 Worker 与领域事件

`apps/worker` 通过消息队列订阅领域事件，用于**登录审计、用户生命周期、以及各业务域扩展**（公司、组织、Agent、协作、记忆、任务、计费、模板等，以 `apps/worker/src/app.module.ts` 与具体监听器为准）。

| 需求 ID | 用户视角 / 系统需求 | 实现要点 | 验收要点 |
|---------|---------------------|----------|----------|
| ASYNC-01 | 用户登录成功或失败后，系统**异步记录**可追溯信息（日志/后续扩展告警）。 | `auth.login_success`、`auth.login_failed` 等队列 | Worker 消费无堆积；日志字段完整。 |
| ASYNC-02 | 用户创建/更新/删除后，下游**异步处理**（审计、同步、自治编排等）。 | `user.created` / `updated` / `deleted` | 事件至少一次投递语义以实现为准。 |
| ASYNC-03 | 其它业务事件（公司、组织、Agent…）由对应 Listener 处理；部分逻辑可能为**最小可运行实现**（结构化日志），便于先验证链路。 | 见 `contracts/events` 与各 worker 模块 | 与 `07-worker-messaging-autonomous.md` 对照。 |

**说明**：终端用户通常不直接调用 Worker HTTP；`/api/worker/*` 路由存在时为内部或受限用途，需在部署层限制暴露。

## 6.2 独立日志服务（可选组件）

若部署 `apps/logging`：

| 需求 ID | 用户故事 | 实现要点 | 验收要点 |
|---------|----------|----------|----------|
| LOG-01 | 应用能将日志**批量上报**到日志服务。 | `POST /api/logs`、`POST /api/logs/batch` | 高吞吐下不丢关键字段。 |
| LOG-02 | 授权方能在开发/测试环境**查询**日志（内存查询实现为主）。 | `GET /api/logs`、`GET /api/logs/:id` | 敏感字段脱敏（password、token 等）。 |

## 6.3 已知边界与待确认项

1. **部分 Service 内部**可能仍有 `TODO` 或占位逻辑；验收以**对外路由与主路径行为**为准，深链路需单独测试用例。
2. **安全中间件**默认对未带 Header 的请求跳过；若要做「强制签名」环境，需在网关配置与客户端约定上显式启用并全链路测试。
3. **多租户 RLS** 与迁移版本强相关；升级迁移后需回归租户隔离用例。
4. **Swagger** 仅在非生产且 `SWAGGER_ENABLED=true` 等条件下开放，不作为对外 SLA 的一部分。

## 6.4 需求追溯建议

- 产品验收：按本系列 **需求 ID** 建立测试用例矩阵。
- 开发对照：路由与 RPC 模式名以 `routes.config.ts`、`rpc-patterns.config.ts` 及 API 模块为准。
- 变更流程：接口变更时同步更新 **`docs/architecture/implemented-features.md`** 与本目录对应章节。
