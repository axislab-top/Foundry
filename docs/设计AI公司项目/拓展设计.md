你的 Foundry 架构 已升级，新增了 RpcModule、Rpc 控制面（ApiRpcProxyService / WebhooksRpcProxyService）和 RPC queues，这对实现 AI 公司工厂 项目非常有利。它允许 Gateway 更高效地通过 RPC（而非纯 HTTP Proxy）调用下游服务，适合强类型、内部控制面操作（如 Agent 执行、任务状态更新）。
下面是基于更新架构的完整设计与实现方案，充分利用现有能力（Gateway 安全/韧性/路由/RPC、事件驱动 Worker、Logging、Storage 等），最小改动即可扩展成商用 SaaS。
1. 整体架构调整（最小侵入式扩展）
在现有 Mermaid 图基础上，新增/增强以下部分：

Gateway：
新增 TenantModule（多租户上下文）。
新增 WebSocketGateway（实时群聊，使用 Socket.io + Redis Adapter，支持动态 rooms）。
扩展 RpcModule：增加更多 RPC patterns（如 company.create、agent.execute、task.dispatch）。
TenantGuard + CompanyContextInterceptor：所有请求/ RPC 自动注入 company_id。

apps/api（核心业务层）：
新增模块（复用现有 Config/DB/Cache/Storage/Guards/Messaging）：
CompaniesModule
OrganizationModule
AgentsModule
SkillsModule
CollaborationModule（群聊元数据 + 历史）
TasksModule
MemoryModule（RAG 向量记忆）
BillingModule
TemplatesModule

扩展 ApiRpcController：暴露 RPC 方法给 Gateway/Worker 调用（如 validateCompanyAccess、getAgentConfig）。

apps/worker（自治大脑）：
增强 MessagingModule：订阅新事件（company.heartbeat、task.needs_execution、agent.need_approval 等）。
新增 AgentExecutionService + HeartbeatService。
使用 libs/ai 执行 LangGraph 流程。

新增共享库（强烈推荐）：
libs/tenant：TenantGuard、CompanyContext、装饰器 @CurrentCompany()。
libs/ai：LangGraph Orchestrator、LLMClient（多模型路由）、ToolRegistry（Skills 转 Tools）、HierarchicalSupervisor。

数据库：
共享 PostgreSQL。
多租户策略推荐：Shared Schema + tenant_id (company_id) + Row Level Security (RLS)（简单、迁移友好、适合起步）。后期若需更强隔离，可平滑迁移到 schema-per-company。
所有核心表（如 agents、tasks、memories、skills）加 company_id 字段 + RLS policy。
使用 PGVector 扩展实现向量记忆（公司/部门/Agent 级）。

事件与 RPC 结合：
HTTP/REST 用于用户交互（客户端 → Gateway → api）。
RPC 用于内部高效调用（Gateway/Worker → api）。
MQ 事件用于异步/自治（company.created → 初始化结构；heartbeat → 自主运行）。


2. 核心模块设计与实现要点
多租户基础（最先实现）

在 Gateway 添加 TenantGuard（解析 JWT 或 header 中的 company_id，校验用户属于该公司）。
使用 CLS（Continuation Local Storage）或 Request Context 传播 company_id。
DatabaseModule 支持动态设置 app.current_tenant（结合 RLS）。
所有 Service 查询默认加 WHERE company_id = :companyId。

CompaniesModule & OrganizationModule

POST /companies：一键创建公司 → 发布 company.created 事件 → Worker 初始化默认董事会、CEO、部门模板（行业推荐）。
组织结构：树形/图结构存储（使用 closure table 或 nested set，或简单 parent_id + order）。
前端用 React Flow 拖拽 → 调用 API 更新汇报线。
RPC 方法：getOrganizationTree(companyId)。

AgentsModule + SkillsModule（实现 OpenClaw-like 大量 Skills）

Agent 实体：id, company_id, name, role, department_id, systemPrompt, llmModel, skills: Skill[]。
Skill 实体：id, company_id?, name, category, toolSchema (JSON), promptTemplate, implementationType ('builtin'|'langgraph'|'api'), permissions。
Skills 分为：平台全局（官方/商城）、公司私有。
Agent 绑定 Skills（多对多）。
执行：libs/ai 的 ToolRegistry 把 Skill 动态转为 LangGraph Tool / function calling。
一键招聘/模板：从 TemplatesModule 导入 Agent + Skills 组合。

CollaborationModule（实时群聊）

Gateway 新增 WebSocketGateway（@WebSocketGateway() + RedisIoAdapter，支持多实例水平扩展）。
房间设计：company:{companyId}（默认用户+CEO）、department:{deptId}、task:{taskId} 等动态加入。
CEO 说“拉工程部” → 服务端动态 join room → 对应 Agent 进入群聊。
支持流式输出（LLM streaming 通过 Socket emit）。
历史存 MemoryModule + 可搜索。

自治运行（Worker + LangGraph）

libs/ai 使用 LangGraph.js Supervisor 实现 hierarchical multi-agent：
顶层：CEO Supervisor（决策风格、性格可配置）。
中层：部门主管 Supervisor。
底层：员工 Agent（执行具体 Skills）。

HeartbeatService：定时（Cron 或 MQ 事件）触发 company.heartbeat → CEO 审查待办、拆解任务、分配、汇总。
任务流程：用户大目标 → TasksModule 记录 → Worker 触发 LangGraph 执行 → Skills 调用 → 结果写回 Memory + Billing。
Human-in-the-loop：Agent emit agent.need_approval 事件 → 通过 WebSocket 推送到用户群聊，用户回复继续。

MemoryModule

使用 PGVector 存储向量嵌入。
分层：公司长期记忆（历史决策、文档）、部门知识库、Agent 短期上下文。
自动总结：Worker 定期把对话/任务总结成向量存入（结合现有 StorageModule 存文件）。
RAG：在 LangGraph 节点中检索相关记忆。

BillingModule

实时记录每个 Agent 调用、Skill 使用、Token 消耗（company_id 隔离）。
模型路由：CEO 用强模型（Claude/GPT-4o），执行 Agent 用便宜模型。
预算上限 + 超支暂停（通过 ResilienceModule 限流或 Worker 检查）。

TemplatesModule

公司模板、Agent 商城（含预绑 Skills）。
一键导入 → 复制配置 + 发布初始化事件。

审计与日志

复用现有 AuditInterceptor + AuditModule：记录所有 Agent 决策、群聊、任务操作。
所有服务上报到 Logging 服务（结构化 JSON，带 company_id、traceId）。

3. 客户端与管理端

用户客户端：Next.js 15（App Router） + React Flow（组织图） + Socket.io-client（实时群聊 + 流式）。
页面：公司列表/创建、仪表盘（任务、预算、Agent 状态）、组织结构编辑、群聊界面、Agent/Skills 管理。

5. 关键技术选型与注意事项

AI 编排：LangGraph.js + Supervisor（支持多层 hierarchical，已有成熟模式）。
实时：Socket.io + Redis Adapter（已匹配你的 CacheModule/Redis）。
向量记忆：PGVector（与现有 DatabaseModule 集成好，支持 RLS）。
RPC vs HTTP：用户交互用 HTTP（Gateway 动态路由），内部 Agent 执行/状态用 RPC（更高效、强类型）。
安全性：敏感 Skills/操作强制 Human-in-the-loop + 审批；所有查询强制 company_id。
成本：Billing 记录每一步消耗；Gateway ResilienceModule 防止滥用。
扩展：未来加真实人类成员 → 扩展 RBAC（复用现有 Roles/PermissionsGuard）。

这个设计高度复用了你当前的 Foundry 架构（Gateway 统一入口 + RPC + 事件循环 + Logging + Storage），新增部分主要集中在 apps/api 新模块、libs/ai 和 WebSocket。
