2. 系统架构建议（2026 最优技术选型 + Trade-off）
推荐架构：Hybrid WebSocket + Event-Driven + Stateful Room Service

实时通道：Socket.IO v4.8+（已支持在你 Gateway 的 CollaborationWsModule 中）
使用 Redis Adapter（@socket.io/redis-adapter）实现多实例水平扩展 + Sticky Session 可选
Namespace: /collaboration
Room 概念直接映射 Socket.IO Room

消息持久化与查询：
主存储：PostgreSQL（JSONB + pgvector 用于语义搜索）
热消息缓存：Redis Streams 或 Redis Sorted Set + Hash（最近 7 天消息）
全文 + 向量搜索：结合 pgvector + LlamaIndex.TS 或自建 Hybrid Search

状态与编排层：
新增 Collaboration Service（可放在 Worker 或独立微服务，但推荐先放在 Worker 中）
使用 LangGraph 或 StateGraph 管理每个 Room 的「会话状态机」（尤其是含 Human-in-the-loop 的审批流）

流式输出实现：
Agent 执行时不一次性返回，而是通过 SSE-like 的 chunk 事件推送到 Socket.IO
使用 socket.emit('message.chunk', { roomId, messageId, chunk, tokenIndex })


Trade-off 分析（我直接毒舌点评）：

纯 Redis Pub/Sub：太轻，缺乏持久化与查询能力 → 放弃
纯 PostgreSQL 轮询：延迟高，用户体验差 → 放弃
Kafka + WebSocket：过于重，运维复杂 → 当前阶段不需要
当前推荐：Socket.IO + Redis Adapter + PG 主存 + Redis 热缓存 → 最平衡，复用你现有设施最高

3.3 实时群聊核心实现（Gateway 侧）
TypeScript// apps/gateway/src/collaboration/collaboration.gateway.ts
@WebSocketGateway({
  namespace: '/collaboration',
  cors: { origin: '*' }, // 生产用 ConfigService
})
export class CollaborationGateway {
  constructor(
    private readonly roomService: RoomService, // 通过 RPC 调用 API
    private readonly messageService: MessageService,
  ) {}

  @SubscribeMessage('joinRoom')
  async handleJoinRoom(
    @MessageBody() { roomId }: { roomId: string },
    @ConnectedSocket() client: Socket,
  ) {
    const canJoin = await this.roomService.canJoin(client.data.userId, roomId);
    if (!canJoin) throw new WsException('无权加入');

    client.join(`room:${roomId}`);
    await this.roomService.addMember(roomId, client.data.userId, 'human');

    this.server.to(`room:${roomId}`).emit('memberJoined', { ... });
  }

  // 关键：流式消息转发
  @SubscribeMessage('sendMessage')
  async handleMessage(
    @MessageBody() dto: SendMessageDto,
    @ConnectedSocket() client: Socket,
  ) {
    // 1. 持久化消息（通过 RPC 到 API）
    const msg = await this.messageService.createMessage(dto);

    // 2. 广播给房间所有人（含 Agent 监听器）
    this.server.to(`room:${dto.roomId}`).emit('newMessage', msg);

    // 3. 如果是 Agent 发送的消息且包含 @user，则触发 Human-in-the-loop
    if (dto.senderType === 'agent' && dto.mentions?.includes('human')) {
      await this.triggerApprovalFlow(dto);
    }
  }

  // Agent 执行流式输出专用通道
  async broadcastStreamChunk(roomId: string, chunk: StreamChunk) {
    this.server.to(`room:${roomId}`).emit('message.chunk', chunk);
  }
}

4. 我主动指出的风险与最优解

风险1：Socket.IO 多实例 sticky session 不佳 → 解：必须用 Redis Adapter + 合理配置 horizontal scaling
风险2：高并发下消息顺序问题 → 解：每条消息带 sequenceNumber + Redis Stream 保证顺序
风险3：Agent 流式输出与人类消息交织混乱 → 解：引入 threadId + parentMessageId + 前端时间线严格排序
风险4：记忆注入导致 Prompt 过长 → 解：在 MessageService 中做智能总结 + 向量检索 TopK 注入
