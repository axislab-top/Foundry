# 可运行数字公司 — 完整计划清单

> **北极星**：Foundry 作为可运行的数字公司 — 目标与任务可编排，执行可追踪，运营可监控，资源有预算，高风险必须经过人的审批闸门；人退居董事会，系统承担从员工到中层的大部分闭环。
>
> 本文档按 **工作流（Workstream）× 阶段** 组织，每条均为可验收交付物。实施建议：**先竖切闭环，再扩大自治**。

## 1. 整盘成功定义

- [ ] **目标—任务—执行** 全链路在系统内可查询（含版本与指派）。
- [ ] **每一次执行** 有 trace（工具调用、外部请求、错误、耗时、关联预算）。
- [ ] **运营状态** 可监控（SLI/SLO、告警、Runbook、状态页或等价物）。
- [ ] **资源** 有预算与硬限制（按公司/项目/代理/任务类型）。
- [ ] **高风险动作** 无法绕过审批（配置、发布、花钱、删数、扩权）。
- [ ] **人** 只在「董事会视图」：审批队列、例外处理、战略变更；日常闭环由系统完成。

## A. 基础与数据模型（阶段 0）

**必须先稳；无此则后续编排与治理无法落地。**

### A1. 身份、租户与权限

- [ ] 公司/租户模型与数据隔离策略落地（RLS 或等价）并写准入测试。
- [ ] 角色模型：`board`（人）、`system`、`agent`、`service account` 分野清晰。
- [ ] 权限矩阵初版：读/写/执行工具/花预算/改策略/发布 分级。
- [ ] API Key / JWT / 内部服务认证边界文档化。

### A2. 核心「公司对象」

- [ ] **Mission / Goal**：公司使命与可度量目标（含时间范围、负责人实体）。
- [ ] **Project / Initiative**：承上启下的容器（对齐路线图与资源池）。
- [ ] **Task / WorkItem**：状态机（待办/进行中/阻塞/待审批/完成/失败）、优先级、依赖（DAG 或至少阻塞关系）。
- [ ] **Assignment**：任务指派给 Agent 或人（人通常只在审批或例外）。
- [ ] **Artifact**：产出物索引（代码变更、文档、报告、发布记录）关联任务。

### A3. 事件与消息契约

- [ ] 领域事件清单：`task.*`、`goal.*`、`approval.*`、`budget.*`、`heartbeat.*`、`execution.*` 等命名与载荷规范。
- [ ] 消费者幂等策略统一（事件 ID、业务幂等键、去重窗口）。
- [ ] 死信队列与失败可见性（失败必须进「可运营」界面，而不只打日志）。

## B. 编排层（阶段 1）

**像公司一样分工、委派与节拍。**

### B1. 组织与委派

- [ ] 组织树 / 汇报关系（谁向谁汇报、谁可拆任务）。
- [ ] 委派规则：默认路由（按技能标签、负载、成本）。
- [ ] 跨团队请求：工单或内部「协作任务」模型。

### B2. 计划与拆解

- [ ] 从 Goal 生成 Project/Task 的模板或半自动流程（可先规则化，再 AI 化）。
- [ ] 任务依赖解析与拓扑排序/阻塞传播。
- [ ] 周期性工作：**心跳调度**（固定间隔、cron、事件触发）与「错过补偿」策略。

### B3. 与现有自治组件对齐（Foundry）

- [ ] 明确接入点：`TaskHeartbeatScheduler`、CEO 心跳图（如 `infrastructure/ai`）、`AutonomousOrchestratorService` 等与统一任务模型的数据源一致。
- [ ] Checkpointer / 长时运行状态：生产级存储与迁移策略（非内存）。
- [ ] 详见：`docs/features/07-worker-messaging-autonomous.md`、`docs/autonomous/`、根目录《项目功能与能力说明》相关章节。

## C. 执行与可追踪（阶段 2）

**没有黑箱；董事会只看 trace 也能问责。**

### C1. 执行运行时

- [ ] Agent 执行适配层：统一入口（HTTP/Bash/IDE agent/内部 worker）与超时、取消、重试策略。
- [ ] **工具白名单**：按角色/任务类型限制可调用的工具与 API。
- [ ] 沙箱或最小权限：文件系统、网络、密钥注入边界。

### C2. Trace / 审计（任务级）

- [ ] 单次 run 的 **RunRecord**：输入摘要、步骤、工具调用、外部请求元数据（URL 脱敏规则）、错误栈引用。
- [ ] 与网关 **Audit**、**Logging** 服务关联：`requestId` / `runId` / `taskId` 贯通。
- [ ] 不可篡改或「仅追加」的审计策略（至少对审批与策略变更加强）。

### C3. 产出物与发布

- [ ] 变更集关联（PR/分支/构建号）与任务绑定。
- [ ] 发布流水线闸门（与 F 阶段联动）与 trace 联动。

---

## D. 运营可监控（阶段 3）

**公司灯火通明；一人公司也要告警分级。**

### D1. 可观测性标准

- [ ] 服务级：健康检查、RED/USE 类指标、错误预算概念。
- [ ] 业务级：任务成功率、平均完成时间、队列积压、心跳迟到率。
- [ ] 统一日志字段规范（`companyId`、`taskId`、`runId`、`agentId`）。

### D2. 告警与 On-call

- [ ] 告警路由：Webhook → IM/邮件/Pager 等价物（Worker 中占位类通知需替换为真通道）。
- [ ] 告警分级：信息 / 警告 / 页面级。
- [ ] Runbook：每条 P1 告警对应一页「先做什么、再做什么、何时升级给人」。

### D3. 产品化运营视图

- [ ] 「董事会仪表盘」：今日风险、预算消耗、待审批、失败任务 Top、系统健康。
- [ ] 状态页或维护公告机制（对外或对内）。

---

## E. 资源与预算（阶段 4）

**钱与算力有闸门；防止「自治」变成「账单自治」。**

### E1. 预算模型

- [ ] 预算维度：公司 / 项目 / 代理 / 任务类型 / 时间周期（月、周）。
- [ ] 计量：token、外部 API 调用次数、GPU 分钟、第三方 SaaS 配额（能拿到什么记什么）。
- [ ] **硬停止**：预算耗尽 → 自动暂停派单/暂停高成本工具（与 `billing.checkAllowance` 等路径对齐并验收）。

### E2. 成本归因

- [ ] 每次 run 写入预估/实际成本（可先粗粒度，后细化）。
- [ ] 成本报表：谁最贵、哪类任务烧钱、哪条链路 ROI 差。

### E3. 采购与「花钱类动作」

- [ ] 任何扣费类操作必须走 **预算检查 + 审批（若超阈值）**。

---

## F. 治理与审批闸门（阶段 5）

**人 = 董事会；高风险在工程上不可绕过。**

### F1. 风险分级（RACI 的工程化）

- [ ] 操作分类：L0 自动、L1 事后审计、L2 事前审批、L3 双人/延迟批准（可选）。
- [ ] 典型 L2/L3：生产配置变更、数据删除、对外发布、权限提升、超预算执行、新代理「入职」。

### F2. 审批产品面

- [ ] 审批队列 UI/API：`pending / approved / rejected / expired`。
- [ ] 审批上下文包：diff、影响面、回滚方案摘要、关联任务与 run trace。
- [ ] 审批与执行 **原子绑定**：未批准不得执行；批准后生成不可伪造的执行令牌。

### F3. 策略与变更管理

- [ ] 策略即配置：版本化、可回滚、可审计。
- [ ] 「紧急制动」：全局暂停、单代理暂停、单项目熔断。

---

## G. 自治扩展（阶段 6）

**中层闭环；从规则到模型渐进。**

### G1. 监督与复盘

- [ ] 中层 supervisor：检查下属产出质量、合并报告、上升异常（先规则，后模型）。
- [ ] 固定复盘任务：日报/周报自动生成并写入记忆或知识库。

### G2. 记忆与知识

- [ ] 决策与教训回灌：失败 run → 结构化条目 → 下次计划引用（与 memory 域衔接）。
- [ ] 与客户/产品相关的知识分区（避免污染与泄露）。

### G3. 多公司与模板（可选）

- [ ] 公司模板导出/导入：组织、策略、技能包、默认预算（注意密钥脱敏）。

---

## H. 安全、合规与生存（全程并行）

### H1. 安全

- [ ] 密钥管理（KMS/环境隔离）、最小权限数据库账号、依赖与镜像扫描节奏。
- [ ] 供应链：第三方 Agent/工具调用审计与版本锁定策略。

### H2. 合规与条款

- [ ] 隐私与数据留存策略（日志/trace 保留期）。
- [ ] 用户协议与数据处理说明（若对外服务）。

### H3. 业务连续性

- [ ] 备份与恢复演练（数据库、checkpoint、关键配置）。
- [ ] 灾难预案：单点故障清单与降级模式。

---

## I. 集成与对外职能（按需）

- [ ] Webhook 出站：审批结果、任务完成、告警（`apps/webhooks` 能力与事件源、模板补齐）。
- [ ] 入站集成：支付、邮件、IM、工单系统（Bring-your-own 亦可）。
- [ ] 文档与开发者门户（降低支持负担）。

## 11. 里程碑与节奏

| 里程碑 | 核心交付 | 验收 |
|--------|----------|------|
| **M1** | 任务模型 + 心跳 + 全链路 ID + 基础仪表盘 | 一条任务从创建到完成可在 UI/API 追踪 |
| **M2** | Run trace + 日志/指标贯通 + 告警到 IM | 故障 10 分钟内定位到 run |
| **M3** | 预算硬限制 + 超支自动停 | 压测证明无法绕过 |
| **M4** | 审批闸门覆盖生产/删数/扩权 | 未批准动作在技术上不可执行 |
| **M5** | Supervisor 复盘闭环 + 记忆回灌 | 重复类失败率下降可度量 |
| **M6** | 模板化多公司/多项目组合（可选） | 新开「公司」分钟级可复制 |

## 12. 使用方式与仓库对照

- **脊梁**：**A + B + C** — 没有则不算「数字公司」运行时。
- **器官**：**D + E + F** — 决定能否像真公司一样持续存活、可审计、可刹车。
- **放大器**：**G + H + I** — 决定自治深度、对外能力与抗风险能力。
- **实施方法**：每一阶段选取 **一条竖切场景**（例如「每日巡检任务」或「发版审批」）贯穿 A–F，避免先做孤立模块。
- **仓库索引**：功能分域见 `docs/features/README.md`；HTTP/服务清单见 `docs/architecture/implemented-features.md`；架构总览见 `架构.md`。
