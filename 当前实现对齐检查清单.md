### 2. 真正的多租户隔离设计目标（2026 顶尖标准）

**核心定义**：  
每个用户（平台租户）的**每家公司（companyId）** 拥有一个**独立、持久、可恢复的隔离云电脑空间**。  
- 数据（记忆、群聊、任务、Artifact）物理/逻辑隔离。  
- 执行（代码、build、deploy、shell 命令）在微虚拟化环境中运行，只能影响本空间。  
- 即使 Agent 被 prompt 诱导执行 rm -rf，也只能自残本空间，且可一键恢复。

**隔离强度要求**（拒绝纯 Docker）：
- 数据库：PostgreSQL RLS + tenant context 强制注入。
- 文件/执行：gVisor（MVP，高密度） + Firecracker microVM（生产主力，内核级隔离）。
- 存储：MinIO 专属 PVC + versioning + snapshot（CoW）。
- 网络：默认 deny-all + 白名单 egress。
- 运行时：非 root 用户、无 sudo、fsRoot=/workspace。

### 4. 关键风险 & 强制验收标准（Cursor 必须验证）

**高优先级风险（必须零容忍）**：
- RLS 配置错误或 tenant context 未注入 → 跨公司数据泄露。
- 执行仍走 Worker 容器直连 → OpenClaw 类事故。
- 无 snapshot → 自残后无法恢复。
- 资源无 quota → 成本爆炸或 noisy neighbor。

**验收标准**（每项必须通过）：
- 任意公司查询只能看到自己数据（跨公司查询返回空）。
- 高危命令默认被拦截或强制审批。
- 执行环境 fsRoot=/workspace，非 root，无法访问宿主机或其他公司 volume。
- 执行后 snapshot 存在，一键恢复有效。
- 全链路 trace 包含 companyId + task.runId + sandboxId。
- 红队测试：模拟恶意 prompt + rm -rf，结果仅影响本空间且可恢复。
