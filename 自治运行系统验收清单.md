## 4. 对照验收清单（核心验收标准）

### 4.1 Heartbeat 调度与自治触发
- [ ] Heartbeat 按配置频率（每小时/每天）稳定触发
- [ ] 每次 Heartbeat 能正确检索公司当前状态（Memory + 未完成任务 + 预算）
- [ ] Heartbeat 可同时响应关键事件（task.completed、budget.warning 等）
- [ ] 系统在无用户干预情况下能持续运行至少 3 个完整 Heartbeat 周期

### 4.2 CEO Hierarchical Supervisor 决策质量
- [ ] CEO Supervisor 能正确分析目标 + Memory + Budget，生成合理的任务拆解树
- [ ] 拆解结果符合公司当前组织结构（OrganizationModule）
- [ ] 性格配置生效（保守型 CEO 更谨慎，激进型更主动）
- [ ] 支持条件决策：预算紧张时自动降级模型或减少并行任务
- [ ] 分层执行正常：CEO → 部门主管 Supervisor → 执行 Agent

### 4.3 执行闭环与一致性
- [ ] 子任务能正确分配给对应 Agent，并触发执行（Skills + Memory RAG）
- [ ] 任务进度实时更新到 TasksModule，状态流转正确
- [ ] 执行结果自动汇总并向上汇报（CEO 生成最终报告）
- [ ] 任务完成后自动生成经验总结并写入 MemoryModule
- [ ] 父子任务状态联动一致（子任务完成 → 父任务进度更新）

### 4.4 Human-in-the-loop 与可控性
- [ ] 关键决策（大额预算、战略变更、高风险任务）能触发 `agent.need_approval` 事件
- [ ] 审批请求通过 WebSocket 实时推送至用户群聊
- [ ] 用户回复后系统能继续执行（审批状态正确流转）
- [ ] 用户可随时干预正在运行的任务（暂停、调整、终止）

### 4.5 成本与预算集成
- [ ] 每次 LLM 调用、Skill 调用都经过 BillingModule 实时记账
- [ ] 模型路由策略生效（CEO 用强模型，执行层用便宜模型）
- [ ] 预算预警和硬限制正常工作（80% 预警、100% 暂停非关键操作）
- [ ] 超支场景下系统能优雅降级而非崩溃

### 4.6 数据隔离、安全与审计
- [ ] 所有自治操作严格在当前 company_id 下执行（RLS + TenantGuard）
- [ ] 跨公司数据完全隔离
- [ ] 完整审计链路：Heartbeat 触发 → 决策 → 执行 → 汇报 全过程可追溯
- [ ] 敏感操作有权限校验和日志记录

### 4.7 非功能验收
- [ ] **稳定性**：系统连续运行 24 小时无崩溃，重启后能从 Checkpoint 恢复
- [ ] **性能**：单个 Heartbeat 周期内完成审查 + 拆解 + 执行的耗时合理
- [ ] **实时性**：进度、汇报、审批请求通过 WebSocket 实时推送
- [ ] **可观测性**：TracingModule 能追踪完整自治链路（含 companyId、traceId）
- [ ] **错误恢复**：执行失败有重试 + DLQ 机制，关键错误能通知用户

## 5. 风险与剩余建议（非阻塞）
- 长周期任务的上下文管理与状态爆炸风险（建议加强 Memory 总结机制）
- Heartbeat 频率与系统成本的动态平衡策略
- 多公司并发 Heartbeat 时的 Worker 资源隔离与限流
- 极端场景测试（预算耗尽、大规模任务树、多个公司同时运行）
- 建议补充压力测试：模拟 10+ 公司同时运行 Heartbeat + 任务执行

## 6. 全链路 E2E 验证场景（必须通过）

3. 执行中触发 Human-in-the-loop → 用户审批 → 继续执行
4. 预算接近上限 → 系统预警 + 自动降级模型
5. 任务完成后经验总结写入 Memory，下一次 Heartbeat 能利用历史知识
