## 后端详细设计（Backend Detailed Design）

本节给出后台管理系统的后端落地细节，覆盖鉴权/RBAC、HTTP/RPC 接口约定、数据模型与迁移、审计日志、运行时接入与可观测性。后端基于现有架构：`gateway (HTTP/WS) + api (业务/RPC) + worker (异步执行) + messaging (事件总线)`。

### 1. 鉴权与权限（RBAC）

1. 管理端调用均要求 `JWT + 管理员角色`，并使用统一的 `RolesGuard/PermissionsGuard` 体系。
2. 数据访问强制租户隔离：通过 `TenantGuard` + `UserContextMiddleware` 注入 `companyId`，并影响 TypeORM 会话级租户上下文（RLS/Session Set）。
3. 审计与追踪：网关层统一写入 `audit_logs`，对敏感字段做脱敏（例如 `secret/key/token/authorization` 等字段名命中规则）。
4. 角色约定（与 UI 一致）：
   - `superadmin/super_admin`：全部可见可操作
   - `admin`：可管理公司/模板/Key 池/商城，不操作系统级设置
   - `auditor`：只读审计日志与监控

### 2. 网关（Gateway）与业务服务（API）通信约定

1. 管理端 HTTP：由 `apps/gateway` 暴露 `GET/POST/PUT/DELETE` 形式的管理 API（统一前缀：`/api/admin/*`）。
2. 管理端 RPC：网关内部通过 `API_RPC_CLIENT` 调用 `apps/api` 的 RPC handler；RPC 入参使用 DTO 校验（`validateRpcDto`）。
3. 运行时（Worker）与计费：Worker 执行 LLM 后通过事件 `billing.consumption.requested` 上报 tokens 与模型信息，由 API 统一落库并更新预算/使用率。

### 3. LLM Keys（大模型 Key 池）后端设计

#### 3.1 目标与核心语义

1. `LLM Keys` 用于 worker 在运行时调用第三方大模型（OpenAI/Anthropic 等）的真实密钥。
2. 配额单位：按 token 计算（输入/输出 token 统计算作 `used_tokens`）。
3. Key pool 范围（与你当前确认的方向对齐）：
   - pool 为“全局池”：按 `provider + model` 过滤可用 key，不做 `company/agent` 绑定。
   - “已分配公司数/使用历史”通过 billing 记录推导（按 `billing_records` 聚合统计）。
4. 安全要求：
   - secret 只在密钥创建/轮换时返回一次；数据库存储加密密文或不可逆形式（本计划推荐使用现有 `EncryptionService` 体系加密存密钥密文）。
   - secret 不允许进入 MQ/RPC payload 或审计日志；worker 获取 secret 仅用于本次调用。

#### 3.2 数据模型（建议表）

1. `llm_keys`（密钥主表）
   - `id`：UUID 主键
   - `provider`：模型提供方（openai/anthropic/azure/...）
   - `model`：模型名（gpt-4o / claude-3-5-sonnet 等）
   - `keyAlias`：key 别名（用于管理端显示）
   - `encryptedSecret`：加密后的 secret（不可直接明文落库）
   - `isActive`：是否启用
   - `dailyQuotaTokens`：每日配额（token）
   - `lastUsedAt`：最后使用时间
   - `createdAt/updatedAt`
2. `llm_key_daily_usage`（每日使用聚合表）
   - 唯一键：`(llm_key_id, usageDate)`
   - `usedTokens`：累计已消耗 token
   - `updatedAt`
3. `billing_records` 扩展字段（用于统计“使用历史/已分配公司数”）
   - 增加 `llm_key_id`（可空，兼容历史数据）
   - 建立索引：`(company_id, llm_key_id, occurred_at)`（用于统计与筛选）

#### 3.3 HTTP 管理接口（Gateway -> API）

管理接口由 `apps/gateway` 暴露，RPC 转发到 `apps/api`：

- `GET /api/admin/llm-keys?provider&model&page&...`
  - 返回 key 列表（provider+model 过滤、分页、状态筛选）
- `POST /api/admin/llm-keys`
  - 创建 key（provider/model/keyAlias + secret + dailyQuotaTokens）
  - secret 仅返回一次（创建结果里包含明文展示给 UI，但日志脱敏）
- `PUT /api/admin/llm-keys/:id`
  - 更新描述、dailyQuotaTokens、启停状态
- `POST /api/admin/llm-keys/:id/rotate`
  - 轮换 secret（保留 id，更新 encryptedSecret；旧 secret 立即失效）
- `POST /api/admin/llm-keys/:id/disable` / `enable`
  - 启用/禁用
- `DELETE /api/admin/llm-keys/:id`
  - 删除 key（或软删，视合规策略）

> 注意：Worker “acquire” 是 RPC 级别内部调用，不暴露给外部。

#### 3.4 RPC 接口（API -> Worker / Gateway）

RPC handler 主要包括：

1. `llmKeys.acquire`
   - 输入：`modelName/provider`（以及可选的策略参数如 taskPriority）
   - 输出：
     - `llmKeyId`
     - `decryptedSecret`
2. `llmKeys.admin.list/create/update/disable/rotate/...`
   - 由网关管理端调用
   - 一律带 actor/permissions 校验

#### 3.5 运行时接入（Worker 执行前获取 Key）

1. Worker 在 CEO/主执行图解析出 `modelName` 后先请求 `llmKeys.acquire`，获取可用 key 与 secret。
2. LLM client 构造由 `ceo-chat-model.factory.ts` 支持传入 apiKey（OpenAI/Anthropic 分支分别使用传入 key，而不是只读环境变量）。
3. LLM 调用完成后获取 usage tokens（input/output 或等价 metadata），把 tokens 与 `llmKeyId` 写入 `billing.consumption.requested` 事件。
4. API 计费入账时把 `llmKeyId` 落入 `billing_records`，并更新 `llm_key_daily_usage.usedTokens`。

#### 3.6 审计与可观测性

1. Web/HTTP 管理接口：网关层审计 `audit_logs` 自动脱敏 key/secret。
2. RPC/Worker 侧：避免在 RPC payload 与 MQ payload 里传播 secret；worker 只在内存构造 LLM client 并调用。
3. 监控指标建议：
   - key 获取成功率（acquire 成功/失败）
   - daily quota 耗尽的 key 命中次数
   - acquire->billing 入账一致性（按 traceId/agent/公司维度抽样核对）

### 4. 与现有计费（Billing）对齐的改造点

当前计费模型以 `billing_records(modelName, inputTokens, outputTokens)` 为核心；为支持 “LLM Key 使用历史与剩余配额”，需要：

1. 在 `billing.consumption.requested` 事件中携带 `llmKeyId`（便于后端落库）。
2. 在 `billing_records` 增加 `llm_key_id` 字段。
3. 增加每日使用聚合表 `llm_key_daily_usage`，用于快速计算今日剩余配额与健康状态。
